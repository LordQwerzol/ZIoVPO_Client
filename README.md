# ZloVPO Client

**Учебный проект по дисциплине "Защита информации от вредоносного программного обеспечения"**

Клиент для проекта [сервера](https://github.com/LordQwerzol/ZIoVPO_Server).

---
## Технологии и требования

- Среда разработки: **Qt Creator/Visual Studio Code**
- Операционная система: **Windows**
- Язык программирования: **C/C++**
- Фреймворк: **Qt6**
- Система сборки: **CMake**
---

##  **Функциональность**

### Задание 1

1. При запуске добавлять иконку в трей.
2. Левая кнопка по иконке → показать главное окно.
3. Правая кнопка → контекстное меню:  
   - «Открыть» → показать главное окно.  
   - «Выход» → завершить приложение.
4. При пересоздании панели задач иконка восстанавливается.
5. Поддерживать запуск без показа главного окна.
6. Закрытие главного окна → приложение уходит в трей (не завершается).
7. В главном окне: меню «Файл» → «Выход» → завершение работы.
8. Только один экземпляр приложения для одного пользователя (именованный мьютекс). При повторном запуске — выход до добавления иконки.
9. Сборка через **MSBuild** или **CMake**. Артефакт: один `.exe` со всеми зависимостями.
### Задание 2
#### Требования к Windows-службе:
1. Служба должна запускать графическое приложение во всех терминальных сессиях (кроме сессии 0). Запускаемое приложение должно выполняться от имени владельца терминальной сессии. Главное окно при запуске должны быть скрыто
2. Служба должна отслеживать входы новых пользователей и запускать в их терминальных сессиях графическое приложение. Запускаемое приложение должно выполняться от имени владельца терминальной сессии. Главное окно при запуске должны быть скрыто.
3. Служба должна отключить обработку сигналов Stop и Shutdown
4. Служба должна запустить сервер Windows RPC. В качестве транспорта следует использовать ALPC. Служба работает до тех пор, пока сервер Windows RPC не будет остановлен
5. Служба должна зарегистрировать для клиентов интерфейс в Windows RPC, с помощью которого клиенты смогут останавливать сервис 
6. При остановке служба должна завершить все запущенные графические приложения
#### Требования к графическому приложению:
1. Графическое приложение при старте должно проверять состояние Windows-службы. Если она остановлена, графическое приложение должно её запустить и дождаться состояния Running, после чего завершить свою работу. 
2. Графическое приложение должно проверять, кто является родительским процессом. Если родительским процессом не является Windows-служба приложение должно завершить свою работу 
3. Пункт Выход в главном меню главного окна должен останавливать Windows-службу 
4. Пункт Выход в контекстном меню иконки в области уведомлений панели задач должен останавливать Windows-службу
### Задание 3
#### Требования к Windows-службе:
1. Служба должна поддерживать аутентификацию пользователя путем отправки HTTPS-запроса определённым методом к определённому эндпоинту Web-сервиса. После успешной аутентификации служба должна сохранить полученные JWT-токены (Access и Refresh) в оперативной памяти.
2. Служба должна периодически обновлять Access и Refresh токены. Период обновления должен рассчитываться исходя из сроков действия токенов. 
3. Служба должна поддерживать выход из учётной записи путём удаления текущих Access и Refresh токенов. 
4. Служба должна поддерживать запрос текущего состояния лицензии путем отправки HTTPS-запроса определённым методом к определённому эндпоинту Web-сервиса. После успешного запроса лицензионный тикет должен быть сохранён в оперативной памяти. 
5. Служба должна периодически обновлять лицензионный тикет. Период обновления должен рассчитываться исходя из данных в лицензионном тикете. 
6. Служба должна поддерживать активацию продукта путём отправки HTTPS-запроса определённым методом к определённому эндпоинту Web-сервиса. После успешного запроса лицензионный тикет должен быть сохранён в оперативной памяти. Если эндпоинт активации не возвращает лицензионный тикет, после активации необходимо запросить текущий статус лицензии. 
7. Служба должна удалить лицензионный тикет при выходе пользователя из аккаунта. 
8. Служба не должна передавать JWT-токены и лицензионные тикеты клиентским приложениям 
9. Служба не должна сохранять JWT-токены и лицензионный тикет в файловой системе 
10. Служба должна зарегистрировать для клиентов интерфейсы в Windows RPC:
    - Для получения информации о текущем аутентифицированном пользователе 
    - Для входа и выхода из учётной записи 
    - Для получения информации об активной лицензии 
    - Для активации продукта
11. При отсутствии лицензионного тикета служба должна прекратить фоновые задачи и возвращать ошибку отсутствия лицензии при запросе через Windows RPC для всех интерфейсов, связанных с функциональностью антивируса.
#### Требования к графическому приложению:
1. Графическое приложение при запуске должно получить информацию о текущем аутентифицированном пользователе. 
2. Если пользователь не аутентифицирован:
   - функциональность антивируса должна быть заблокирована; 
   - приложение должно показать форму аутентификации.
3. В случае неуспешной аутентификации:
   - функциональность антивируса должна быть заблокирована; 
   - пользователю должна быть выдана ошибка; 
   - вновь показана форма аутентификации.
4. Если пользователь уже аутентифицирован, или успешно прошёл аутентификацию, приложение должно:
   - показать имя (логин) пользователя на главном экране; 
   - запросить статус лицензии.
5. Если приложение не имеет лицензии:
   - функциональность антивируса должна быть заблокирована;
   - на главном окне должна отображаться форма ввода кода активации. 
6. В случае неуспешной активации:
   - функциональность антивируса должна быть заблокирована;
   - должна быть отображена ошибка;
   - форма активации показана повторно.
7. Если приложение имеет лицензию или успешно прошло активацию:
   - функциональность антивируса разблокирована;
   - на главном экране должна отобразится дата истечения лицензии.
8. Графическое приложение должно либо периодически опрашивать, либо быть подписано на изменение состояния лицензии. При изменении статуса лицензии главный экран должен показывать соответствующую информацию
